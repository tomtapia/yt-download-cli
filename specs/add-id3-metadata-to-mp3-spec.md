# Specification for ID3 Metadata in MP3 Output

## Goal
Add ID3 metadata support to MP3 files generated by `yt-download-cli` so that older players, car stereos, and basic media library software can display useful track information after download.

## Context
The CLI already supports `--mode audio --container mp3` by downloading a YouTube audio source and transcoding it with `ffmpeg`. Today the resulting `.mp3` file contains audio only, without embedded metadata. The new behavior should write common ID3 tags during MP3 generation using metadata already available from the download pipeline.

## Requirements
1. **Target Format**
   - Only apply ID3 metadata when the final output container is `mp3`.
   - Do not change `m4a`, `webm`, or video output behavior in this iteration.

2. **ID3 Version**
   - Write ID3v2.3 tags for maximum compatibility with older playback devices and software.
   - Do not require ID3v1 support in v1 unless it is effectively free through the chosen implementation path.

3. **Minimum Metadata Fields**
   - Title: YouTube video title
   - Artist: channel/author name when available
   - Album: optional, leave unset in v1 unless a reliable source is available
   - Comment: optional, may be used for source attribution such as the original YouTube URL
   - Track number, year, genre, lyrics, chapters, and cover art are out of scope for the first iteration unless a clean implementation path already exists.

4. **Source of Metadata**
   - Use the existing `VideoMetadata` gathered during download resolution.
   - Prefer stable values already exposed by the current pipeline over adding new scraping logic just for tags.

5. **Implementation Path**
   - Prefer adding tags during the existing `ffmpeg` MP3 transcode step instead of introducing a second file-rewrite pass.
   - If `ffmpeg` requires explicit metadata arguments, populate them there.
   - Ensure the output remains a valid `.mp3` even when some metadata fields are missing.

6. **Failure Behavior**
   - Missing optional metadata must not fail the download.
   - If metadata writing fails as part of MP3 generation, surface a clear error rather than silently producing a partially broken file.

## Expected CLI Behavior
- `yt-download-cli download <url> --mode audio --container mp3` should produce an `.mp3` with readable title and artist metadata when available.
- No new CLI flags are required in v1 for metadata control.

## Testing
- Unit tests should verify the metadata arguments passed to the MP3 `ffmpeg` invocation.
- Integration-oriented tests should confirm:
  - MP3 output still works without metadata values such as missing author
  - non-MP3 outputs do not start receiving ID3-specific behavior
  - the title and artist fields are mapped from the existing metadata model

## Constraints
- Keep the solution compatible with the current TypeScript + `ffmpeg` architecture.
- Do not add YouTube-specific metadata scraping beyond what the downloader already resolves.
